# 알고리즘 파이프라인 후보 라이브러리

## 문서 목적
이 문서는 실거래형 트레이딩 시스템 개발을 위한 **대규모 파이프라인 후보 저장소 문서**이다. 목적은 단순히 모델 아이디어를 몇 개 적어두는 것이 아니라, 실제 연구·구현·백테스트·페이퍼트레이딩으로 이어질 수 있는 **고품질 파이프라인 후보군을 구조적으로 축적**하는 데 있다.

이 문서는 특히 다음 목표를 가진다.

1. 모델 아이디어를 산발적으로 떠올리지 않고 체계적으로 보존한다.
2. `모델 패밀리 × 데이터 입력 × 타깃 × 포트폴리오/실행 방식`의 조합 공간을 명확히 정의한다.
3. 조합 공간 전체를 무작정 다 구현하는 것이 아니라, 실전 가치가 높은 조합을 우선적으로 선별한다.
4. 연구 단계에서 탈락한 파이프라인도 문서에 남겨 중복 시도를 줄인다.
5. 코드 구현 전에 먼저 문서 차원에서 후보군을 충분히 확보한다.

이 문서는 다음 네 개의 핵심 모델 패밀리를 중심으로 파이프라인 후보를 정리한다.

1. **베이지안 접근**
2. **SDE 접근**
3. **강화학습(RL) 접근**
4. **기존 금융 통계 방식 + 딥러닝 결합 접근**

---

# 1. 사용 원칙

## 1.1 왜 Cartesian Product 방식이 필요한가
실전 퀀트 시스템에서는 좋은 결과가 우연히 몇 개의 모델에서만 나오지 않는다. 오히려 다음과 같은 조합적 탐색이 필요하다.

- 어떤 **모델 패밀리**가
- 어떤 **시장 입력 데이터**와 결합할 때
- 어떤 **예측 타깃**에 대해
- 어떤 **포트폴리오/리스크/실행 구조**와 함께
가장 안정적인지 탐색해야 한다.

따라서 이 문서는 단일 모델 중심이 아니라 다음 조합 공간을 기준으로 작성된다.

```text
Pipeline = Model Family × Model Variant × Input Bundle × Target × Allocation/Execution Layer
```

다만 모든 Cartesian Product를 전부 구현하는 것은 비효율적이므로, 문서에서는 다음 원칙으로 **선별된 고품질 파이프라인 후보**를 저장한다.

- 경제적/금융적 의미가 있는 조합
- 데이터 누수 없이 구현 가능한 조합
- 백테스트/페이퍼트레이딩으로 검증 가능한 조합
- 실전 운영 관점에서 지나치게 취약하지 않은 조합

# 3. 평가용 메타데이터 템플릿

모든 파이프라인은 아래 메타데이터를 문서에 기록한다.

- Pipeline ID
- 이름
- 모델 패밀리
- 핵심 모델
- 입력 번들
- 예측 타깃
- 포트폴리오/실행
- 기대 장점
- 예상 약점
- 구현 난이도
- 데이터 요구수준
- 실전 적합성
- 우선순위

구현 난이도는 다음과 같이 분류한다.
- L1: 매우 쉬움
- L2: 쉬움
- L3: 중간
- L4: 어려움
- L5: 매우 어려움

실전 적합성은 다음처럼 기록한다.
- H: 초기 실전 후보
- M: 연구 우선 후보
- R: 연구 전용 / 고위험

# 8. 각 파이프라인의 구현 전 체크 항목

문서에 있다고 바로 구현하지 않는다. 각 파이프라인은 구현 전에 아래 항목을 다시 검토해야 한다.

## 8.1 데이터 가능성
- 필요한 입력 데이터가 실제로 확보 가능한가
- point-in-time 정합성을 보장할 수 있는가
- 텍스트/옵션/미시구조 데이터의 latency는 감당 가능한가

## 8.2 비용/운영 가능성
- 거래 빈도가 지나치게 높지 않은가
- 백테스트가 비용에 취약하지 않은가
- 실거래에서 운영 가능한 단순성인가

## 8.3 리스크 적합성
- tail risk를 어떻게 통제할 것인가
- 포지션 사이징 기준이 명확한가
- kill switch 위에 자연스럽게 올라갈 수 있는가

## 8.4 구현 난도 대비 가치
- 지금 구현해야 하는가
- 더 단순한 baseline이 먼저 필요한가
- 패밀리 내 유사 후보와 중복되지 않는가

# 11. 결론

이 문서는 트레이딩 시스템 개발을 위한 **대규모 파이프라인 아이디어 저장소**이며, 단순 아이디어 메모가 아니다. 핵심은 다음과 같다.

1. 모델 패밀리를 네 축으로 나눴다.
2. 입력 데이터 번들을 구조화했다.
3. 타깃과 포트폴리오/실행 레이어까지 조합 공간에 포함했다.
4. Cartesian Product의 정신을 유지하되, 실제 가치 있는 조합만 선별했다.
5. 최소 100개 이상, 실제로는 120개의 파이프라인 후보를 문서화했다.
6. 바로 구현하면 안 되고, 우선순위 shortlist를 기준으로 단계적으로 구현해야 한다.

가장 중요한 철학은 다음 한 줄로 요약된다.

**좋은 모델 몇 개를 우연히 찾는 것이 아니라, 조합 공간을 체계적으로 문서화하고 그 위에서 가장 실전적인 파이프라인을 선별하는 것이 장기적으로 더 강한 시스템을 만든다.**
