# 실거래 트레이딩 시스템 전체 검증 체크리스트 v2

이 문서는 **실제 자동매매 시스템이 가능한 한 실패 없이 동작하도록 만들기 위한 최고수준의 전체 체크리스트**이다. 단순한 개발 TODO가 아니라, 연구·구현·백테스트·페이퍼트레이딩·실거래·모니터링·장애복구까지 모두 포함한 운영 검증 문서다.

핵심 목표는 다음 네 가지다.

1. **재현 가능성**: 어느 서버에서나 같은 방법으로 설치되고 실행되어야 한다.
2. **검증 가능성**: 각 단계가 통과되었는지 자동으로 확인할 수 있어야 한다.
3. **운영 안정성**: 실거래에서 잘못된 주문, 중복 주문, 초과 노출, 데이터 오류를 막아야 한다.
4. **직접 구현 최소화**: 가급적 검증된 기존 패키지를 사용하고, 직접 구현은 반드시 필요한 부분만 한다.

이 문서는 체크리스트 형식이지만, 각 항목에 바로 적용 가능한 기준과 권장 패키지, 실패 신호, 통과 조건까지 채워 넣었다.

# 0. 절대 원칙

## 0.1 시스템 철학
- [ ] **직접 구현보다 검증된 패키지 우선** 원칙을 문서 최상단에 명시했다.
  - 기준: 공식 문서 품질, 유지보수 상태, GitHub 활동성, 사용자 수, 테스트 존재 여부, 운영 사례
  - 이유: 퀀트 시스템은 아이디어보다 운영 실수로 망하는 경우가 훨씬 많기 때문
- [ ] 연구용 코드와 운영용 코드를 분리했다.
  - 연구용: 노트북, 실험 스크립트, 가설 검증
  - 운영용: 주문, 리스크, 계좌 동기화, 장애 처리
- [ ] 예측 모델과 주문 집행 시스템을 분리했다.
  - 예측이 틀려도 시스템이 망가지지 않도록 하기 위함
- [ ] 백테스트 성능과 실거래 가능성을 별도 통과 기준으로 정의했다.
- [ ] 실거래 이전에 반드시 **백테스트 → 워크포워드 검증 → 페이퍼트레이딩 → 소액 실거래 → 단계적 확대** 순서를 강제한다.
- [ ] 기본 모드는 항상 `dry_run` 또는 `paper_trading`이다.
- [ ] 리스크 엔진은 어떤 전략보다 우선한다.
- [ ] 모든 주문은 리스크 엔진과 계좌 상태 체크를 통과해야만 전송된다.
- [ ] 운영 중 이상 징후가 하나라도 발견되면 자동으로 노출 축소 또는 주문 중단이 가능해야 한다.

## 0.2 가장 큰 실패 원인 사전 차단
- [ ] 미래 정보 누수(data leakage) 방지 체계를 갖췄다.
- [ ] 백테스트 비용/슬리피지/유동성 가정을 실거래보다 낙관적으로 잡지 않았다.
- [ ] 실거래 키와 테스트 키를 절대 혼용하지 않는다.
- [ ] 동일 종목 중복 주문, 반대 주문 충돌, stale signal 재사용을 방지한다.
- [ ] 타임존과 장 운영시간을 명확히 관리한다.
- [ ] 데이터 공급 실패 시 조용히 빈 데이터로 넘어가지 않고 즉시 실패 처리 또는 fallback 처리한다.

## 0.3 실패 신호
- [ ] “백테스트가 좋으니 바로 실거래해도 된다”는 식의 판단이 들어가 있지 않다.
- [ ] 한 개의 메인 스크립트에서 수집, 학습, 예측, 주문, 로그를 전부 처리하지 않는다.
- [ ] notebook에서 되면 운영에서도 될 것이라고 가정하지 않는다.

# 1. 패키지 선택 체크리스트

## 1.1 패키지 선정 정책
- [ ] 모든 핵심 기능에 대해 “직접 구현 vs 기존 패키지 사용” 비교표를 만들었다.
- [ ] 기존 패키지를 우선 채택할 때 다음 항목을 평가했다.
  - [ ] 공식 문서가 충분한가
  - [ ] 최근 6~12개월 내 업데이트가 있는가
  - [ ] Python 버전 호환성이 명확한가
  - [ ] 테스트가 존재하는가
  - [ ] 운영 사례가 있는가
  - [ ] 치명적 이슈가 방치되어 있지 않은가
- [ ] 의존성 충돌 가능성이 큰 패키지는 미리 샌드박스에서 설치 검증했다.
- [ ] 버전은 고정 또는 범위 제한을 걸었다.
- [ ] “좋아 보이는 새 패키지”보다 “검증된 안정 패키지”를 우선했다.

## 1.2 권장 패키지 스택

### 데이터 처리
- [ ] 기본 DataFrame 계층을 `pandas` 또는 `polars` 중 하나로 통일했다.
  - 권장: 호환성 중심이면 `pandas`
  - 권장: 대규모 배치 성능이 중요하면 `polars`
- [ ] 수치 계산은 `numpy`를 사용한다.
- [ ] 시계열/통계 계산은 `scipy`, `statsmodels`를 우선한다.

### 머신러닝
- [ ] 베이스라인 ML은 `scikit-learn`으로 통일했다.
- [ ] 부스팅은 `LightGBM` 또는 `XGBoost`를 사용한다.
- [ ] 실무 기본 추천은 다음과 같다.
  - 빠른 탐색: `LightGBM`
  - 보수적 비교: `XGBoost`
- [ ] 전처리/파이프라인은 `scikit-learn Pipeline` 또는 명확한 자체 파이프라인으로 관리한다.

### 딥러닝
- [ ] 딥러닝 프레임워크는 `PyTorch`로 통일했다.
- [ ] 훈련 루프 복잡도가 크면 `PyTorch Lightning`을 검토한다.
- [ ] 실험 추적과 체크포인트 저장은 프레임워크 기능 또는 `MLflow/W&B`와 연동한다.

### 강화학습
- [ ] RL은 직접 구현보다 `stable-baselines3` 기반으로 시작한다.
- [ ] 환경 인터페이스는 `gymnasium` 형식으로 맞춘다.
- [ ] RL은 실거래 첫 단계의 핵심 엔진이 아니라, 후순위 실험 대상으로 둔다.

### 포트폴리오 최적화
- [ ] 최적화는 `cvxpy`를 기본 계층으로 삼는다.
- [ ] solver는 `OSQP`, `ECOS`, `SCS` 중 호환성을 확인한다.
- [ ] MVO, Min-Var, CVaR, 제약 최적화는 가급적 `cvxpy` 기반으로 일관성 있게 구현한다.

### 백테스트
- [ ] 빠른 연구용: `vectorbt`
- [ ] 이벤트 기반 검증: `backtrader` 또는 자체 이벤트 엔진
- [ ] 두 방식 중 적어도 하나는 주력, 하나는 교차 검증용으로 유지한다.

### 설정 및 검증
- [ ] 설정 검증은 `pydantic` 또는 `pydantic-settings`를 사용한다.
- [ ] 복잡한 실험 구성은 `Hydra`를 고려한다.

### 실험 추적
- [ ] `MLflow` 또는 `Weights & Biases` 중 하나를 표준으로 정했다.
- [ ] 데이터 버전, 모델 버전, git commit hash, config hash를 저장한다.

### 브로커
- [ ] 브로커는 공식 SDK 또는 널리 쓰이는 래퍼 우선
  - IBKR: 공식 API 또는 `ib_insync` 계열
  - Alpaca: 공식 SDK
- [ ] 브로커 SDK를 그대로 쓰지 않고, 내부 추상화 계층 뒤에 숨긴다.

### 로깅 / 모니터링
- [ ] 표준 `logging` 또는 `structlog/loguru` 중 하나로 통일한다.
- [ ] 로그 포맷은 JSON 또는 명확한 텍스트 포맷으로 고정한다.

## 1.3 패키지 남용 방지
- [ ] 같은 기능을 하는 패키지를 여러 개 동시에 쓰지 않는다.
  - 예: pandas + dask + modin + polars를 모두 쓰지 않음
- [ ] 너무 많은 래퍼를 겹겹이 씌우지 않는다.
- [ ] 핵심 경로는 디버깅 가능한 단순한 구조로 유지한다.

## 1.4 통과 조건
- [ ] 신규 서버에서 단일 설치 스크립트로 환경이 세팅된다.
- [ ] 샘플 파이프라인이 끝까지 실행된다.
- [ ] dependency conflict가 없다.
- [ ] GPU/CPU 두 모드 중 적어도 하나가 완전히 동작한다.

## 4.4 통과 조건

- [ ] secret 없는 상태에서 안전하게 실패

# 5. 데이터 수집 및 정합성 체크리스트

## 5.1 공급원 관리
- [ ] 모든 데이터 공급원에 대해 목적, 지연, 비용, 라이선스를 문서화했다.
- [ ] 가격 데이터와 기업행동(corporate actions)을 함께 관리한다.
- [ ] 벤치마크 데이터와 개별 종목 데이터를 분리 저장한다.
- [ ] 재무, 뉴스, 거시 데이터는 공개 시점 기준으로 정렬 가능하다.

## 5.2 정합성 검증
- [ ] 날짜가 오름차순이다.
- [ ] 인덱스 중복이 없다.
- [ ] 음수 가격/음수 거래량이 없다.
- [ ] 비정상 급등락은 플래그 처리한다.
- [ ] adjusted / unadjusted 데이터 혼용 여부를 검사한다.
- [ ] split/dividend 반영 여부가 명확하다.
- [ ] 상장폐지 종목이 조용히 사라지지 않는다.
- [ ] 심볼 변경 이력이 관리된다.

## 5.3 포인트 인 타임(Point-in-Time)
- [ ] 재무 데이터는 발표일 기준으로 사용한다.
- [ ] 뉴스 데이터는 기사 발행 시점 이후에만 사용한다.
- [ ] 애널리스트 추정치 변화는 실제 공개 시점 이후에만 반영한다.

# 6. 피처 엔지니어링 체크리스트

## 6.1 피처 정의
- [ ] 각 피처의 이름, 수식, 필요 입력, 윈도우 길이, lag를 문서화했다.
- [ ] 모든 피처는 미래 정보 없이 계산된다.
- [ ] rolling 계산은 현재 시점 이하만 참조한다.
- [ ] cross-sectional rank는 같은 시점 단면 내에서만 계산한다.
- [ ] 표준화/정규화 기준이 학습과 추론에서 일치한다.

## 6.2 누수 방지
- [ ] 타깃 값이 피처에 섞이지 않는다.
- [ ] 전체 데이터 통계량으로 과거 데이터를 정규화하지 않는다.
- [ ] train/val/test를 나눈 뒤 scaler를 fit한다.
- [ ] 라벨 생성 시 미래 수익률과 현재 시점 피처가 정확히 정렬된다.

## 6.3 안정성
- [ ] 결측치 처리 정책이 피처별로 정의돼 있다.
- [ ] 이상치 clipping/winsorization 여부를 문서화했다.
- [ ] 피처 계산 실패 시 무음 통과하지 않는다.
- [ ] 피처 컬럼 순서가 고정된다.
- [ ] 학습 시점과 추론 시점의 피처 스키마가 일치한다.

## 6.4 저장 및 버전
- [ ] 피처 세트마다 버전이 있다.
- [ ] 피처 생성 config를 저장한다.
- [ ] 학습에 사용한 피처 목록을 artifact로 남긴다.

## 6.5 통과 조건
- [ ] 피처 생성 후 shape 검증 통과
- [ ] NaN 비율 임계치 이하
- [ ] train/test schema mismatch 없음

# 27. 가장 추천하는 “안전한 기본 조합”

실패 확률을 최소화하려면 첫 버전은 아래 조합이 가장 현실적이다.

## 추천 패키지 조합
- 데이터: `pandas`, `numpy`, `scipy`, `statsmodels`
- ML: `scikit-learn`, `lightgbm`, `xgboost`
- DL: `torch`
- 최적화: `cvxpy`, `osqp`
- 백테스트: `vectorbt` + 필요시 `backtrader`
- 설정: `pydantic-settings`
- 실험 추적: `MLflow` 또는 `W&B`
- 브로커: `ib_insync` 또는 공식 API / Alpaca 공식 SDK
- 테스트: `pytest`
- 로깅: `logging` 또는 `structlog`

## 추천 첫 전략 구조
- 시장 레짐 필터: 규칙 기반 + 간단한 HMM
- 예측 엔진: `LightGBM` 또는 `XGBoost`
- 포트폴리오: Equal Weight baseline → Risk Parity → HRP 순서
- 실행: 단순 limit/market hybrid
- 리스크: position cap + daily loss cap + drawdown kill switch

이 조합이 좋은 이유는,
- 너무 많은 딥러닝/강화학습 불확실성을 초기에 제거하고,
- 검증된 전통 스택으로 운영 안전성을 먼저 확보할 수 있기 때문이다.

# 28. 최종 결론

“한 치의 실패 없이”라는 목표를 실제에 최대한 가깝게 만들려면, 다음 순서를 절대 어기면 안 된다.

1. **검증된 패키지 우선**
2. **환경 재현성 확보**
3. **데이터 정합성 검증 자동화**
4. **베이스라인 전략부터 구축**
5. **백테스트와 비용 반영 검증**
6. **리스크 엔진을 전략보다 우선 배치**
7. **페이퍼트레이딩으로 E2E 확인**
8. **소액 실거래 후 단계적 확대**
9. **모니터링·장애복구·롤백 준비**

즉, 최고의 모델이 아니라 **최고로 덜 망가지는 시스템**을 먼저 만들어야 한다. 이 문서는 그 기준으로 작성된 전체 체크리스트다.
